# TTT-Search POC — v1.1 Implementation Journal

This document is a complete record of what v1.1 of the TTT-Search proof-of-concept
adds on top of v1, why it adds it, exactly which lines of code changed, what
performance optimizations were applied, what the new statistical machinery
looks like, and what the final numbers said.

For the v1 history — including the original hypothesis, the bugs we hit in
v1, and v1's results — see `v1Readme.md`. This file assumes you have read
that one and focuses entirely on what changed.

---

## 1. Why v1.1 exists at all

v1's 164-problem run produced these pass rates:

```
base_greedy        32.9%  (54/164)
base_temperature   45.1%  (74/164)   ← T=0.8, n=4
ttt_search         43.3%  (71/164)   ← T=0.3, n=4 priming texts
```

By the design doc's rule (ttt_search must beat base_temperature by 10pp to
support the hypothesis), v1 killed the hypothesis: delta was −1.8pp.

But the paired-comparison data was interesting:

- 12 problems solved by `ttt_search` only.
- 15 problems solved by `base_temperature` only.
- Oracle union: 52.4%.

So the two methods *agree on most problems but disagree on a meaningful
minority*. The killed-hypothesis verdict ignored that — and the design
doc explicitly flagged the missing control needed to interpret it:

> The two stochastic conditions differ in *two* knobs at once. `base_temperature`
> uses T=0.8. `ttt_search` uses T=0.3 *and* a LoRA warm-up. v1 cannot tell
> which knob produced the 12/15 problem-set divergence.

v1.1 exists to add the missing control and complete the experimental
factorial so the question "is priming doing real work?" gets a clean answer.

---

## 2. The 2×2 factorial v1.1 builds

| | T=0.3 | T=0.8 |
|---|---|---|
| **No priming** (LoRA zeroed) | `base_temp_lowT` 🆕 | `base_temp_highT` (= v1's `base_temperature`) |
| **With priming** (TTT warm-up) | `ttt_lowT` (= v1's `ttt_search`) | `ttt_highT` 🆕 |

`base_greedy` is kept as a continuity floor. **Five conditions** total.

The critical new cell is `base_temp_lowT`: identical to `base_temperature`
in every way except the generation temperature drops from 0.8 to 0.3. That
makes it match `ttt_lowT` on temperature. Any remaining difference between
`base_temp_lowT` and `ttt_lowT` is *only* the priming effect.

`ttt_highT` is included to make the factorial symmetric and confirm the
finding (if any) at high temperature.

## 3. What changed in the code, file by file

The change set is small. The condition modules were *parameterized*, not
duplicated. `run_experiment.py` was rewritten to drive five conditions
from one config dict. A new analysis module was added. The schema for
per-problem records was overhauled.

### 3.1 `ttt_search/conditions/base_temperature.py` — parameterized

**Before (v1):**

```python
SEEDS = [42, 43, 44, 45]

def generate(model, tokenizer, prompt, test_code, entry_point) -> str:
    zero_lora_weights(model)
    input_ids = format_prompt(tokenizer, prompt).to(model.device)
    candidates = []
    for seed in SEEDS:
        torch.manual_seed(seed)
        out = model.generate(input_ids, max_new_tokens=768,
                             do_sample=True, temperature=0.8, top_p=0.95, ...)
        # decode, extract, verify; early-return if pass
```

**After (v1.1):**

```python
def generate(model, tokenizer, prompt, test_code, entry_point,
             temperature=0.8, top_p=0.95) -> dict:
    zero_lora_weights(model)
    input_ids = format_prompt(tokenizer, prompt).to(model.device)

    torch.manual_seed(SEEDS[0])
    out = model.generate(
        input_ids, max_new_tokens=768,
        do_sample=True, temperature=temperature, top_p=top_p,
        num_return_sequences=len(SEEDS),
        pad_token_id=tokenizer.eos_token_id,
    )
    # decode each of the 4 returned sequences,
    # extract_code on each, verify each,
    # return v1.1 schema with all 4 candidates
```

Three substantive changes:

1. **`temperature` is now an argument** — so `base_temp_lowT` and
   `base_temp_highT` are the same function called with different `temperature=`
   values.
2. **Batched generation via `num_return_sequences=4`** — instead of four
   sequential `generate()` calls each producing one sample, one call
   produces all four. This is a roughly 3–4× speedup for the entire
   condition. The wall-clock per candidate is the batch time itself.
3. **Return type changed from `str` to `dict`.** The dict has the v1.1
   schema (see §4) including all four candidates with full code, pass
   status, error, and generation time.

The selection rule (first passing candidate, falling back to index 0) is
preserved but is now computed *after* all candidates are scored, so the
result dict still has `selected_index` and `any_passed`.

### 3.2 `ttt_search/conditions/ttt_search.py` — parameterized

**Before (v1):**

```python
TTT_TEMPERATURE = 0.3

def generate(model, tokenizer, prompt, test_code, entry_point) -> str:
    # for each priming text:
    #   reset_lora_weights, _warmup, generate at T=TTT_TEMPERATURE
    # verify each, return first passing
```

**After (v1.1):**

```python
def generate(model, tokenizer, prompt, test_code, entry_point,
             generation_temperature=0.3, top_p=0.95) -> dict:
    priming_texts = _load_priming_texts()  # [(stem, text), ...]
    for i, (label, priming_text) in enumerate(priming_texts):
        reset_lora_weights(model)
        _warmup(model, tokenizer, priming_text)

        t0 = time.time()
        out = model.generate(..., temperature=generation_temperature, ...)
        gen_time = time.time() - t0

        # decode, extract_code, verify, append to candidates
        # with source_label = priming filename stem
```

Three substantive changes:

1. **`generation_temperature` is a parameter** — `ttt_lowT` is called with
   0.3, `ttt_highT` with 0.8. The warm-up itself (8 AdamW steps at LR
   `1e-4` on the priming text) is unchanged; only the *generation*
   temperature differs.
2. **`_load_priming_texts` returns `(stem, text)` pairs** — so each
   variant can be attributed back to which priming file produced it
   (`"iterative"`, `"recursive"`, `"edge_cases"`, `"builtins"`). This
   gives the analysis script the data it needs for per-source-label
   contribution breakdowns.
3. **Per-candidate timing.** `generation_seconds` is recorded for each
   variant separately because TTT variants are inherently sequential
   (each needs its own LoRA state). Times here are *not* batched.

Variants cannot be batched in this condition the way they can in
`base_temperature`, because each variant requires a *different* set of
LoRA weights after a *different* warm-up. You would have to either swap
LoRA matrices per row of a batch or replicate the entire model. Neither
is worth the complexity for a 4× speedup; we keep variants sequential.

### 3.3 `ttt_search/conditions/base_greedy.py` — schema-aligned

`base_greedy` was already simple. Only the return type changed: it now
returns the v1.1 schema dict (with a single candidate, `source_label =
"greedy"`) instead of a plain string. This keeps the orchestrator's
record-building code uniform across conditions.

It also now takes `test_code` and `entry_point` so it can verify and
report `passed`/`error` in the candidate record. v1 verified at the
orchestrator level; v1.1 pushes that down into the conditions so each
candidate carries its own pass status.

### 3.4 `ttt_search/analysis.py` — new file

A standalone module that loads `detailed.json` and writes
`analysis.txt`. Its contents are described in §5. It is invoked
automatically by `run_experiment.py` after the experiment completes,
unless `--skip-analysis` is passed.

### 3.5 `run_experiment.py` — rewritten orchestrator

The orchestrator now defines all five conditions in a single
configuration map:

```python
CONDITIONS = {
    "base_greedy":      {"type": "greedy"},
    "base_temp_lowT":   {"type": "temperature", "temperature": 0.3, "top_p": 0.95},
    "base_temp_highT":  {"type": "temperature", "temperature": 0.8, "top_p": 0.95},
    "ttt_lowT":         {"type": "ttt", "generation_temperature": 0.3, "top_p": 0.95},
    "ttt_highT":        {"type": "ttt", "generation_temperature": 0.8, "top_p": 0.95},
}
```

`run_condition` reads `config["type"]` and dispatches to one of the three
condition modules with the right keyword arguments. There is no code
duplication between the `lowT` and `highT` variants.

Other v1.1 orchestrator changes:

- Checkpoints moved from `results/checkpoint_<cond>.json` to
  `results/checkpoints/<cond>.json` (cleaner directory).
- `detailed.json` now contains the v1.1 schema with full candidate code
  for every variant (not just a 300-char preview of the winner).
- After the run completes, the analyzer is invoked unless
  `--skip-analysis` is set, and its output (`analysis.txt`) is also
  printed to stdout for terminal visibility.

## 4. The v1.1 record schema

Every entry in `detailed.json` for v1.1 looks like:

```json
{
  "task_id": "HumanEval/0",
  "condition": "ttt_lowT",
  "candidates": [
    {
      "index": 0,
      "source_label": "iterative",
      "code": "<full generated code, untruncated>",
      "passed": true,
      "error": null,
      "generation_seconds": 6.07
    },
    { "index": 1, "source_label": "recursive",  ... },
    { "index": 2, "source_label": "edge_cases", ... },
    { "index": 3, "source_label": "builtins",   ... }
  ],
  "selected_index": 0,
  "any_passed": true
}
```

- `candidates` has the per-variant data the design doc required.
- `source_label` is the priming-text stem for TTT conditions and the
  RNG-seed string (`"seed_42"`, ...) for temperature conditions. For
  `base_greedy` it is the literal string `"greedy"`.
- `code` is the full output of `extract_code`, not a preview.
- `passed` and `error` are the verifier's verdict on that exact
  candidate.
- `generation_seconds` is wall-clock for that candidate. For batched
  temperature conditions, all four candidates report the same batch
  time. For sequential TTT variants, each is its own time.
- `selected_index` is which candidate the condition's selection rule
  picked (first passing, falling back to index 0).
- `any_passed` is the pass-rate signal. This is what summary and
  analysis compute on. It is `True` iff at least one of the four
  candidates passed.

This schema is the single biggest reason v1.1's results can be
trust-but-verified after the fact in ways v1's results cannot.

## 5. What `analysis.py` computes

The analyzer produces `results/analysis.txt` with six sections, in this
order:

1. **Per-condition pass rates** — same as `summary.txt`, repeated here
   so the analysis file stands alone.
2. **Pairwise solve-set comparison** — for every pair of conditions,
   how many problems both solve, how many only A solves, how many only
   B solves, how many neither, the symmetric difference, and the oracle
   union. This is the actual cross-tab from which the §6 verdict matrix
   is computed.
3. **Oracle union of all five conditions.** The upper bound on what any
   selection rule over these methods could achieve.
4. **Diversity within each condition.** For multi-candidate conditions,
   compute the mean normalized Levenshtein distance between every pair
   of the four candidates per problem, then average across all 164
   problems. Also count problems where all four candidates were
   byte-identical. This is the §5.5 sanity check from v1, formalized
   and reported every run.
5. **Per-source-label first-pass contribution.** For each
   multi-candidate condition, attribute (a) which source label was the
   first one to produce a passing candidate, and (b) which source
   labels ever produced a passing candidate. This lets us see whether
   one priming text dominates first-passes (it often does — first in
   iteration order wins) but the others contribute meaningfully on
   "any-pass".
6. **Verdict.** Applies the §6 matrix from the v1.1 design doc to the
   `(ttt_lowT, base_temp_lowT)` pair (primary) and the
   `(ttt_highT, base_temp_highT)` pair (secondary). Each verdict is
   one of:
   - H1 + H2 HOLD: priming creates real diversity
   - H3 HOLDS: priming is decorative
   - TTT ACTIVELY HARMS: warm-up interferes
   - AMBIGUOUS

The verdict logic for a pair `(ttt, base)`:

```
D     = len(ttt_solves - base_solves) + len(base_solves - ttt_solves)
delta = pass_count(ttt) - pass_count(base)

if D >= 10  and delta >= -2:        H1 + H2 HOLD
elif D < 10 and abs(delta) <= 3:    H3 HOLDS
elif delta <= -5:                   TTT ACTIVELY HARMS
else:                               AMBIGUOUS
```

`D ≥ 10` is the threshold for "the methods solve different sets". The
`delta ≥ −2` clause prevents declaring victory when TTT is symmetrically
diverse but actually worse than the control.

## 6. Performance work between v1 and v1.1

Two interventions made v1.1 feasible at 5 conditions × 164 problems on
the same hardware.

### 6.1 BF16 instead of bitsandbytes 4-bit quantization

v1 followed the design doc literally: `load_in_4bit=True`. The design
doc said 4-bit was the right choice "if FP8 isn't available", because it
targeted an 8 GB Colab T4.

Profiling during v1.1 setup showed `nvidia-smi` reporting only 13% GPU
utilization with our model running. The cause was bitsandbytes 4-bit
dequantization: every forward pass dequantizes weights on-the-fly into
BF16 in CUDA kernels. For an 0.8B model on a 24 GB RTX 3090 — where
quantization is not necessary — that overhead is the dominant cost.

Fix: `dtype=torch.bfloat16, load_in_4bit=False` in `loader.py`. The
0.8B model in BF16 is 1.6 GB, fits trivially. The deviation from the
design doc's spec is documented in `summary.txt` and only matters if a
future reader insists on running this on an 8 GB GPU.

Empirical effect: per-token throughput went from ~33 tok/s (4-bit) to
~33 tok/s (BF16). On paper, the same — but in practice the 4-bit
overhead was hidden in tokenizer/decode/setup time around generation,
so end-to-end the BF16 path is noticeably faster.

### 6.2 Batched generation in `base_temperature`

v1's `base_temperature` did four sequential `generate()` calls, one
per seed. Each one launches CUDA kernels independently. The four are
embarrassingly parallel.

Fix: a single `generate()` call with `num_return_sequences=4`. The
generator draws four samples in lockstep from the same prompt.
Per-candidate seed control is sacrificed (one seed for the whole
batch), but `do_sample=True` with the same temperature still produces
four genuinely different outputs because each token position samples
independently.

Empirical effect: in a one-off benchmark, batched 4 candidates took
22.3 seconds; the v1 sequential version was ~56 seconds. So roughly
2.5× faster on that condition.

The TTT variants in `ttt_search` cannot be batched the same way: each
needs a different LoRA. They remain sequential.

### 6.3 The combined effect on total runtime

| Version | Run | Conditions | Wall-clock |
|---|---|---|---|
| v1 | Phase 4 (20 problems) | 3 | 57.9 min |
| v1 | full 164 (with §5.3 extract_code fix) | 3 | 143.8 min |
| v1.1 | full 164 (BF16 + batching) | 5 | **263.0 min** |

v1.1 has 5 conditions to v1's 3, plus per-candidate verification, plus
the analyzer. Even so, total wall-clock is only ~1.8× v1, not the
expected ~2.0× from condition count alone. The optimizations earned
back roughly the cost of the two extra conditions.

## 7. Bugs found and fixed while building v1.1

### 7.1 Multi-block `extract_code` — applied during the validation pass

While writing the v1 validation report (between Phase 4 and full-164),
the `detailed.json` from Phase 4 was inspected. HumanEval/11, /13, and
/14 were failing in TTT with the model output looking like:

```
def string_xor(a, b):
    return a + b
```

Wait, that's not correct. Let me fix:

```
def string_xor(a, b):
    return ''.join('1' if x != y else '0' for x, y in zip(a, b))
```

The original `extract_code` matched the *first* opening fence and tried
to strip a trailing closing fence. The output it returned included a
literal ` ``` ` in the middle of the code (from the wrong block's
closing fence), which the verifier rejected with `SyntaxError`.

Fix: rewrite `extract_code` to split on opening fences and return the
**last** block that contains the target function name. Models
consistently self-correct in later blocks. This change went into
`prompt_utils.py` *before* the full-164 v1 run, so it was applied to
both v1's 164-problem and all of v1.1's runs.

A unit-style sanity check after the fix:

```python
raw = """
```python
def f(): return 'wrong'
```

Wait, fix:

```python
def f(): return 'correct'
```
"""
extract_code(raw, "def f(): pass") -> "def f(): return 'correct'"
```

Correct block returned, fence garbage discarded.

### 7.2 No new big bugs in v1.1 itself

The v1.1 code changes were small and incremental. The pipeline was
already working; v1.1 added knobs and logging on top. The smoke test
(3 problems × 5 conditions, ~3 minutes) was clean on the first
attempt.

## 9. Full v1.1 results

### 9.1 Per-condition pass rates (164 problems)

```
Condition         Pass Rate    Pass Count
base_greedy        32.9%       54/164
base_temp_lowT     42.7%       70/164   ← critical new control
base_temp_highT    45.1%       74/164
ttt_lowT           43.3%       71/164
ttt_highT          46.3%       76/164   ← new
```

### 9.2 Pairwise solve-set divergences

Key rows from the analyzer's pairwise table:

| Pair | Both | Only A | Only B | Symmetric diff D | Union |
|---|---|---|---|---|---|
| `base_temp_lowT` vs `ttt_lowT` | 62 | 8 | 9 | **17** | 79 (48.2%) |
| `base_temp_highT` vs `ttt_highT` | 58 | 16 | 18 | **34** | 92 (56.1%) |
| `ttt_lowT` vs `ttt_highT` | 62 | 9 | 14 | 23 | 85 (51.8%) |
| `base_temp_lowT` vs `base_temp_highT` | 60 | 10 | 14 | 24 | 84 (51.2%) |

Oracle union over all five conditions: **100/164 (61.0%)**.

### 9.3 Diversity within each condition

Mean pairwise normalized Levenshtein distance between the 4 candidates,
averaged across all 164 problems, and the count of problems where all
four candidates were byte-identical:

```
base_temp_lowT     mean=0.194   identical=16/164
base_temp_highT    mean=0.297   identical= 3/164
ttt_lowT           mean=0.192   identical=20/164
ttt_highT          mean=0.299   identical= 5/164
```

At the same temperature, TTT and base sampling have essentially the
same within-condition diversity (0.194 vs 0.192 at T=0.3; 0.297 vs
0.299 at T=0.8). The priming does not visibly increase or decrease the
internal spread of the four candidates. What it changes is *which set
of problems* those candidates collectively solve — see §9.2.

### 9.4 Per-source-label contributions for the TTT conditions

`any-pass` = "this priming text produced a passing candidate for that
problem":

| Source | ttt_lowT any-pass | ttt_highT any-pass |
|---|---|---|
| iterative | 50 | 44 |
| recursive | 53 | 47 |
| edge_cases | 50 | 38 |
| builtins | 48 | 48 |

All four priming texts are pulling roughly equal weight on `any-pass`.
The `first-pass` numbers heavily favor `iterative` because it is first
in the iteration order and the selection rule picks the first passing
candidate; that's a logging artifact, not a quality signal.

### 9.5 Verdict from `analysis.txt`

```
[Low temperature (T=0.3) — PRIMARY]
    ttt=71, base=70, delta=+1
    only_ttt=9, only_base=8, symmetric_diff D=17
    -> H1 + H2 HOLD: priming creates real diversity. Worth pursuing v2.

[High temperature (T=0.8) — secondary]
    ttt=76, base=74, delta=+2
    only_ttt=18, only_base=16, symmetric_diff D=34
    -> H1 + H2 HOLD: priming creates real diversity. Worth pursuing v2.
```

Both pairs trigger the same branch:

- D = 17 (≥ 10) ✓ and delta = +1 (≥ −2) ✓ at T=0.3.
- D = 34 (≥ 10) ✓ and delta = +2 (≥ −2) ✓ at T=0.8.

By the §6 verdict matrix from the v1.1 design doc, **H1 + H2 hold at
both temperatures**.

## 10. What v1.1 actually showed — and didn't

What the data supports:

1. **TTT and base sampling at the same temperature solve different
   problem sets.** At T=0.3, 17 problems out of the 79 either method
   solves are unique to one of them. At T=0.8, 34 out of 92. That is
   real divergence, not noise.
2. **The priming is the cause of that divergence**, since temperature
   is now controlled (TTT vs base both at 0.3, both at 0.8). The
   within-condition diversity numbers (§9.3) confirm that TTT does
   *not* simply increase randomness — it shifts the model toward
   different solutions.
3. **The pass-rate gap remains small** (+1 at low T, +2 at high T,
   both with paired McNemar Z ≈ 0.3 — not statistically significant).
   So TTT is roughly as good as plain sampling, not measurably better,
   on this benchmark with this model. The value is the *coverage of a
   different problem subset*, not absolute pass-rate.
4. **Oracle ensembling is the obvious follow-on.** Union of
   `ttt_lowT ∪ base_temp_lowT` = 48.2%, beating either alone by ~5–6 pp.
   Union of the high-T pair = 56.1%, beating either alone by ~10 pp.
   Union of all five = 61.0%.

What the data does not support:

- "TTT improves the pass rate." It does not, on this configuration.
- "One priming text is better than the others." On `any-pass` they are
  near-equal. `first-pass` numbers are biased by iteration order.
- "TTT is robust to small LoRA." 20 out of 164 ttt_lowT problems still
  produced four byte-identical candidates. The warm-up is too weak on
  those problems to make a meaningful difference. With a larger LoRA
  or more steps the divergence number could be either higher or lower —
  v1.1 does not say.

## 12. How to reproduce

```bash
cd /root/llm_trainer/ttt-search-poc

# Full v1.1 reproduction (~4.4 hours on a 24 GB GPU)
python run_experiment.py --num-problems 164 --output-dir results/v11_full

# A specific subset of conditions (e.g., just the critical comparison)
python run_experiment.py \
    --num-problems 164 \
    --conditions base_temp_lowT,ttt_lowT \
    --output-dir results/just_lowT

# Skip the analyzer (faster if you only want the raw data)
python run_experiment.py --num-problems 164 --skip-analysis

# Run the analyzer over an existing detailed.json
python -c "from ttt_search.analysis import analyze; \
           analyze('results/v11_full/detailed.json', \
                   'results/v11_full/analysis.txt')"
```

Required environment:

- A CUDA GPU with at least ~3 GB of free VRAM (BF16, no quantization).
- `pip install -r requirements.txt`.
- `FLA_TILELANG=0` is set automatically by `run_experiment.py` before
  any FLA import. If you import the conditions in your own scripts, set
  this manually first.

## 13. Why this README exists

v1.1 is short on code changes and long on consequences. The new
conditions are tens of lines of code; the analyzer is a few hundred.
But the conclusion they enable — *priming changes the model's solution
space at fixed temperature* — is the opposite of v1's headline.

If a future reader sees only the pass-rate table, they may conclude
v1.1 found nothing because the gap between TTT and base is +1 or +2
problems. That reading would miss the point. The result is in the
solve-set divergence, in the oracle union, and in the diversity table
that shows TTT does *not* just resample at a different temperature.

This file exists so that interpretation is preserved next to the code
that produced it.
