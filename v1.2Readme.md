# TTT-Search POC — v1.2 Implementation Journal

This document is the complete record of v1.2: what question it was built to
answer, why the question matters, every code change, the analyzer's decision
logic, the smoke test, the full run, the final numbers, and the verdict the
data forced.

For prior context — the original hypothesis, the v1 implementation and bugs,
the v1.1 factorial and per-variant logging — see `v1Readme.md` and
`v1.1Readme.md`. This file assumes you have read both.

---

## 1. The state of the question after v1.1

v1.1's analyzer printed **"H1 + H2 HOLD: priming creates real diversity"** at
both temperatures. The full-run table:

| | T=0.3 | T=0.8 |
|---|---|---|
| Base sampling | 70/164 | 74/164 |
| TTT | 71/164 (Δ=+1) | 76/164 (Δ=+2) |
| Sym diff (TTT vs base) | 17 | 34 |
| Within-condition mean Levenshtein | 0.194 vs 0.192 | 0.297 vs 0.299 |

The "H1 + H2 HOLD" verdict was triggered by `D ≥ 10` (at T=0.3, D=17; at T=0.8,
D=34). But the threshold was set in a vacuum. Two specific things in the data
were not interrogated:

- **TTT and base sampling had nearly identical within-condition diversity at
  matched temperature.** If priming were genuinely steering each of the four
  variants toward a different style (iterative / recursive / defensive /
  builtins-ish), you would expect TTT's four candidates to cluster more
  tightly around their priming text — i.e., a *lower* mean pairwise
  Levenshtein. Instead, TTT's spread (0.194 at T=0.3, 0.299 at T=0.8) is
  indistinguishable from base sampling's (0.192, 0.297). That is not what
  "priming bites" looks like.

- **The D=17 symmetric difference had no reference point.** We did not know
  what D you would get from two runs of `base_temp_lowT` with *different
  seed sets*. If that number is also ~17, then TTT's "different solve set"
  is just seed-shuffling noise wearing a priming-shaped costume. If it is
  ~5, then priming actually moves the model meaningfully.

v1.2 exists to measure the noise floor and answer one practical follow-up:
**at matched compute, does mixing methods beat using the best single method?**

---

## 2. The two research questions

### Q1 — Noise floor

> What is the symmetric difference between two runs of the same sampling
> method at the same temperature with different seed sets, on the same 164
> problems?

We call this `D_noise`. Then:

- If `D_ttt − D_noise` is large and positive, TTT really does solve a
  meaningfully different problem set.
- If `D_ttt ≈ D_noise`, TTT is doing nothing the seed could not.
- If `D_ttt < D_noise`, TTT is *converging* the variants (worse than seeds).

The decision-rule numbers, fixed in the v1.2 design doc §4 *before* looking
at any data:

| Condition | Verdict label |
|---|---|
| `D_ttt ≥ D_noise + 8` | priming is real |
| `|D_ttt − D_noise| ≤ 5` | priming is noise |
| `D_ttt ≤ D_noise − 5` | priming reduces diversity |
| else | ambiguous |

Low-temperature is the canonical comparison because at T=0.3 the random
sampling contributes less "free" diversity, so any priming signal is cleaner.

### Q2 — Ensemble payoff at matched compute

> If we had a budget of 8 candidates per problem instead of 4, would
> *mixing* two methods beat *doubling up* on the best single method?

We compute pass rates for five n=8 unions over the existing per-variant
candidates (no new generation, just oracle-union over what we already
sampled). The verdict comparison is `mixed_temp_n8` vs the better of
`pure_lowT_n8` and `pure_highT_n8`.

| Condition | Verdict label |
|---|---|
| `mixed_temp_n8 ≥ best_pure + 4` | mixed helps |
| `|mixed_temp_n8 − best_pure| ≤ 2` | mixed doesn't help |
| else | ambiguous |

Combining Q1 and Q2 produces one of four final verdict lines defined in
§10 of the v1.2 design doc.

## 3. The two new conditions v1.2 generates

Just two — and they are not new code paths, just new *parameter settings*
of conditions that already exist in v1.1:

| New condition | Implementation |
|---|---|
| `base_temp_lowT_seedB` | `base_temperature.generate(...)` with `temperature=0.3` and `seeds=[46,47,48,49]` |
| `base_temp_highT_seedB` | same with `temperature=0.8` |

These are run on the full 164-problem HumanEval set with the same model
(`unsloth/Qwen3.5-0.8B`), the same priming-text-irrelevant LoRA setup
(LoRA is zeroed in these conditions), the same verifier, same chat
template, same `FLA_TILELANG=0`, same BF16 / no-quantization choice.
The only difference between `base_temp_lowT_seedB` and v1.1's
`base_temp_lowT` is the four seeds.

v1.2 does **not** generate new TTT runs. The design doc lists optional
`ttt_lowT_run2` / `ttt_highT_run2` conditions for measuring TTT's own
internal noise, but they are gated on Q1 returning "priming is real." Q1
did not, so those conditions were never run.

## 4. Code changes, in detail

The change set is small. v1.2 is roughly 60 lines of new code plus a
~200-line analyzer.

### 4.1 `ttt_search/conditions/base_temperature.py` — seeds are now a parameter

**Before (v1.1):**

```python
SEEDS = [42, 43, 44, 45]

def generate(model, tokenizer, prompt, test_code, entry_point,
             temperature=0.8, top_p=0.95) -> dict:
    zero_lora_weights(model)
    ...
    torch.manual_seed(SEEDS[0])
    out = model.generate(..., num_return_sequences=len(SEEDS), ...)
    ...
    candidates.append({
        "index": i,
        "source_label": f"seed_{SEEDS[i]}",
        ...
    })
```

**After (v1.2):**

```python
DEFAULT_SEEDS = [42, 43, 44, 45]

def generate(model, tokenizer, prompt, test_code, entry_point,
             temperature=0.8, top_p=0.95,
             seeds: list[int] | None = None) -> dict:
    zero_lora_weights(model)
    seeds = seeds or DEFAULT_SEEDS
    ...
    torch.manual_seed(seeds[0])
    out = model.generate(..., num_return_sequences=len(seeds), ...)
    ...
    candidates.append({
        "index": i,
        "source_label": f"seed_{seeds[i]}",
        ...
    })
```

Three substantive details:

1. The module-level `SEEDS` constant is renamed `DEFAULT_SEEDS` and the
   function takes a `seeds` argument. The default preserves v1.1's exact
   behavior, so the existing `base_temp_lowT` and `base_temp_highT`
   conditions in v1.1 still produce byte-for-byte identical outputs.
2. The `source_label` now reflects the *actual* seeds passed in. After
   v1.2, candidates from `base_temp_lowT_seedB` show
   `source_label="seed_46"`, etc. This means anyone reading
   `detailed.json` can attribute each candidate to its seed without
   knowing which condition it came from.
3. `torch.manual_seed(seeds[0])` is the only seeding call. The four
   candidates in the batch share a single RNG state — the same RNG
   choice we made in v1.1 — but moving `[0]` to a different value
   (46 vs 42) gives a *different starting RNG state* and thus a
   genuinely different sample set. This is the entire mechanism by
   which the v1.2 "seedB" runs differ from the v1.1 "seedA" runs.

### 4.2 `run_experiment.py` — two new entries in `CONDITIONS`

The orchestrator's condition registry got two additions:

```python
CONDITIONS = {
    ...
    "base_temp_lowT_seedB": {
        "type": "temperature",
        "temperature": 0.3,
        "top_p": 0.95,
        "seeds": [46, 47, 48, 49],
    },
    "base_temp_highT_seedB": {
        "type": "temperature",
        "temperature": 0.8,
        "top_p": 0.95,
        "seeds": [46, 47, 48, 49],
    },
}
```

The existing `base_temp_lowT` and `base_temp_highT` entries were also
given an explicit `"seeds": [42, 43, 44, 45]` field so the registry
documents both seed sets in one place rather than relying on the
function's default.

The dispatcher in `run_condition` was updated to pass the seeds through:

```python
elif config["type"] == "temperature":
    result = base_temperature.generate(
        model, tokenizer, prompt, test_code, entry_point,
        temperature=config["temperature"], top_p=config["top_p"],
        seeds=config.get("seeds"),
    )
```

No other code in `run_experiment.py` changed. The same checkpoint logic,
the same `detailed.json` writer, the same summary printer — all reused.

### 4.3 `analyze_v12.py` — new standalone script at project root

This is the bulk of v1.2's code. It lives at the project root (not in
the `ttt_search/` package) because the design doc invocation is:

```bash
python analyze_v12.py
```

— standalone, with its own CLI. Behavior:

1. **Load two `detailed.json` files.** Defaults are
   `results/v11_full/detailed.json` and `results/v12_full/detailed.json`.
   Both paths can be overridden with `--v11` and `--v12`.
2. **Merge by condition name.** v1.1's file contributes `base_greedy`,
   `base_temp_lowT`, `base_temp_highT`, `ttt_lowT`, `ttt_highT`. v1.2's
   file contributes `base_temp_lowT_seedB`, `base_temp_highT_seedB`.
   The merge is a single-line dict union; nothing has to be re-keyed
   because condition names are unique across the two files.
3. **Compute solve sets.** For each condition, the set of task IDs
   where `any_passed == True`. The `any_passed` flag is itself
   computed at experiment time by checking whether *any* of that
   condition's four candidates passed the verifier.
4. **Q1 calculations:**
   - `D_ttt_lowT  = |ttt_lowT_solves Δ base_temp_lowT_solves|`
   - `D_ttt_highT = |ttt_highT_solves Δ base_temp_highT_solves|`
   - `D_noise_lowT  = |base_temp_lowT_solves Δ base_temp_lowT_seedB_solves|`
   - `D_noise_highT = |base_temp_highT_solves Δ base_temp_highT_seedB_solves|`
5. **Apply Q1 decision rule** to `(D_ttt_lowT, D_noise_lowT)` and to
   `(D_ttt_highT, D_noise_highT)` separately.
6. **Q2 calculations.** Each "n=8 union" is computed as
   "union over both source conditions of any task_id where any
   candidate in either condition's record passed the verifier." The
   five combinations are:

   | Variable | Definition |
   |---|---|
   | `pure_lowT_n8`   | `lowT_seedA ∪ lowT_seedB` |
   | `pure_highT_n8`  | `highT_seedA ∪ highT_seedB` |
   | `mixed_temp_n8`  | `lowT_seedA ∪ highT_seedA` |
   | `mixed_ttt_temp_n8` | `ttt_lowT ∪ highT_seedA` |
   | `mixed_ttt_only_n8` | `ttt_lowT ∪ ttt_highT` |

7. **Apply Q2 decision rule.** `best_pure = max(pure_lowT_n8, pure_highT_n8)`.
   Verdict comes from `(mixed_temp_n8, best_pure)` per the §5 thresholds.
8. **Compute the final verdict line** by combining Q1 (canonical =
   T=0.3) and Q2 through the branch logic in `final_verdict()`. The
   four possible output lines are exactly the strings from §10 of the
   v1.2 design doc; nothing is invented.
9. **Write `results/v12_full/analysis_v12.txt`** (path overridable
   with `--out`). Print the same text to stdout for terminal visibility.

The branch logic in `final_verdict()`:

```python
if q1_canonical == "priming is real":
    if q2_mixed_vs_pure == "mixed helps":
        if mixed_ttt_temp >= mixed_temp + Q2_MIXED_HELPS:
            return "VERDICT: Build ensemble system, keep TTT as one source."
        else:
            return "VERDICT: Build ensemble system, drop TTT."
    else:
        return "VERDICT: TTT has real signal but ensembling at n=8 doesn't capture it — needs design work."
elif q1_canonical == "priming is noise":
    if q2_mixed_vs_pure == "mixed helps":
        return "VERDICT: Build ensemble system, drop TTT."
    else:
        return "VERDICT: Whole direction dead. Just sample more."
elif q1_canonical == "priming reduces diversity":
    if q2_mixed_vs_pure == "mixed helps":
        return "VERDICT: Build ensemble system, drop TTT."
    else:
        return "VERDICT: Whole direction dead. Just sample more."
else:
    return "VERDICT: TTT has real signal but ensembling at n=8 doesn't capture it — needs design work."
```

This matrix is hard-coded *before* the analysis runs. There is no
adjustment based on what the numbers turn out to be.

## 5. The thresholds, in one table

All four decision-rule constants live at the top of `analyze_v12.py`,
not buried inside the logic:

```python
Q1_REAL_GAP     = 8   # D_ttt >= D_noise + 8         -> priming is real
Q1_NOISE_BAND   = 5   # |D_ttt - D_noise| <= 5       -> priming is noise
Q1_COLLAPSE_GAP = 5   # D_ttt <= D_noise - 5         -> priming reduces diversity
Q2_MIXED_HELPS  = 4   # mixed_temp_n8 >= max_pure+4  -> mixed helps
Q2_TIE_BAND     = 2   # |mixed_temp - max_pure| <= 2 -> mixed doesn't help
```

If anyone wants to argue with the verdict, the right place to argue is
at these numbers — not at the analysis or the run. They were set in the
design doc and are reproduced verbatim here so the assumption is visible.

## 6. What stayed identical to v1.1

To preserve direct comparability of every number:

- **Model.** `unsloth/Qwen3.5-0.8B`, BF16, no quantization.
- **Inner text tokenizer.** Unwrapped via `tokenizer.tokenizer`.
- **Chat template.** `apply_chat_template(..., enable_thinking=False)`.
- **`max_new_tokens=768`** for generation.
- **`extract_code`.** Same multi-block-aware extractor that picks the
  last code block containing the function definition.
- **Verifier.** Same subprocess + 10s timeout + compile/entry-point
  pre-flight.
- **HumanEval range.** Problems 0–163, loaded the same way.
- **Priming files.** Bit-identical to v1 and v1.1; hashes printed in
  every `summary.txt`.
- **LoRA config.** Same `r=4`, same target modules. Zeroed in the
  `base_*` conditions including the new `seedB` ones.
- **`FLA_TILELANG=0`** set before any FLA import in `run_experiment.py`.
- **Batched generation via `num_return_sequences=4`** in
  `base_temperature`. The seedB conditions inherit this — they batch
  four candidates per problem just like seedA.

The only knob that changes for `base_temp_lowT_seedB` and
`base_temp_highT_seedB` is `seeds[0]` (46 vs 42). Everything downstream
of that — the kernel calls, the temperature, the sampler, the verifier
— is byte-identical.

## 7. The smoke test

Before launching the full 164-problem run, the new conditions and the
analyzer were exercised on 3 problems:

```bash
python run_experiment.py \
    --num-problems 3 \
    --conditions base_temp_lowT_seedB,base_temp_highT_seedB \
    --output-dir results/v12_smoke \
    --skip-analysis
```

Confirmed:

- Both conditions ran end-to-end without crashing.
- `detailed.json` contained 6 records (3 × 2). Each record had four
  candidates. Each candidate's `source_label` read `"seed_46"`,
  `"seed_47"`, `"seed_48"`, or `"seed_49"` — not the v1.1 default.
- Full candidate code was preserved (~650 chars per candidate).
- Per-condition checkpoints landed in `results/v12_smoke/checkpoints/`.

Then the analyzer was pointed at the v1.1 full-run results + the smoke
v1.2 results to confirm the pipeline:

```bash
python analyze_v12.py \
    --v11 results/v11_full/detailed.json \
    --v12 results/v12_smoke/detailed.json \
    --out  results/v12_smoke/analysis_v12.txt
```

The numbers it produced were nonsense (the noise floor was reported as
~70 because v11 had 164 problems and the smoke v12 had 3, so most of
the "only in v11" problems are problems the smoke v12 simply does not
include). But the analyzer completed without errors and printed a
final verdict line, confirming the pipeline was end-to-end correct.

## 9. Full v1.2 results

### 9.1 Per-condition pass rates (combined v1.1 + v1.2)

```
base_greedy                           54/164  (32.9%)
base_temp_lowT  (seedA, T=0.3)        70/164  (42.7%)
base_temp_lowT_seedB (T=0.3)          65/164  (39.6%)   ← v1.2
base_temp_highT (seedA, T=0.8)        74/164  (45.1%)
base_temp_highT_seedB (T=0.8)         70/164  (42.7%)   ← v1.2
ttt_lowT  (gen T=0.3)                 71/164  (43.3%)
ttt_highT (gen T=0.8)                 76/164  (46.3%)
```

Note: `seedB` runs are *slightly* worse than `seedA` runs at both
temperatures. That is within-noise variation. The point of `seedB` is
not its pass rate — it is the symmetric difference against `seedA`.

### 9.2 Q1 — Noise floor measurement

```
D_ttt_lowT   = sym_diff(ttt_lowT,  base_temp_lowT)         = 17
D_noise_lowT = sym_diff(base_temp_lowT, base_temp_lowT_seedB) = 15

D_ttt_highT  = sym_diff(ttt_highT, base_temp_highT)         = 34
D_noise_highT= sym_diff(base_temp_highT, base_temp_highT_seedB) = 36
```

Apply the §4 decision rule:

- At T=0.3: `|D_ttt − D_noise| = |17 − 15| = 2 ≤ 5` → **priming is noise**
- At T=0.8: `|D_ttt − D_noise| = |34 − 36| = 2 ≤ 5` → **priming is noise**

Both temperatures agree. There is no need to fall back to low-T as
canonical.

In plain English: TTT's "different solve set" from v1.1 is entirely
explained by the fact that two runs of base sampling with different
seeds disagree by about the same amount. The priming texts are not
moving the model anywhere the random number generator could not move
it on its own.

### 9.3 Q2 — Ensemble payoff at n=8

```
pure_lowT_n8        = lowT_seedA ∪ lowT_seedB      =  75/164  (45.7%)
pure_highT_n8       = highT_seedA ∪ highT_seedB    =  90/164  (54.9%)
mixed_temp_n8       = lowT_seedA ∪ highT_seedA     =  84/164  (51.2%)
mixed_ttt_temp_n8   = ttt_lowT  ∪ highT_seedA      =  87/164  (53.0%)
mixed_ttt_only_n8   = ttt_lowT  ∪ ttt_highT        =  85/164  (51.8%)
```

`best_pure = max(75, 90) = 90`.

`mixed_temp_n8 = 84`, which is **6 problems below `best_pure`**. The
§5 rule says: `|mixed_temp − best_pure| = 6` is outside both the
"helps" band (≥+4) and the "tie" band (≤±2). The Q2 verdict is
therefore **ambiguous**, but only in the labeled sense — the
substantive observation is that **mixing temperatures is *worse*
than just doubling up on T=0.8**.

The strongest n=8 strategy in the dataset is `pure_highT_n8`:
**90/164 = 54.9%**, achieved by simply running 8 temperature samples
at T=0.8 with two different seed sets.

Adding TTT into a mix (`mixed_ttt_temp_n8 = 87`) does not catch up
to `pure_highT_n8` either. And `mixed_ttt_only_n8 = 85` (two TTT
runs at different temperatures) is also below `pure_highT_n8`.

### 9.4 Final verdict line

The decision logic in `final_verdict()`:

- `q1_canonical` = "priming is noise" (from low-T)
- `q2_mixed_vs_pure` = "ambiguous" (mixed doesn't beat pure with the
  required +4 margin; the difference is in fact −6)

This routes to the `priming is noise → else` branch, returning:

> **VERDICT: Whole direction dead. Just sample more.**

That is what `results/v12_full/analysis_v12.txt` prints as its final line.

## 10. What v1.2 actually closes

The original hypothesis — "TTT-search solves more problems than naive
temperature resampling" — was killed by pass-rate in v1, resurrected by
solve-set divergence in v1.1, and now killed conclusively by v1.2 once
solve-set divergence is compared against its proper noise floor. The
12/15 "TTT only" / "base only" split in v1 that motivated v1.1 is
indistinguishable from two-seed-set sampling variance.

Beyond the headline verdict, three subordinate findings come out of
this run:

1. **Pure temperature sampling at T=0.8 dominates at n=8.** 90/164 from
   8 plain seeds. Mixing in low-temperature samples drops the count to
   84. Mixing in TTT keeps you at 85–87. None of the alternatives beat
   "just sample more at T=0.8."
2. **Within-condition diversity is purely a temperature phenomenon.**
   At T=0.3, mean pairwise Levenshtein is ~0.19 regardless of TTT
   priming. At T=0.8, it is ~0.30 regardless of TTT priming. The
   priming does not change this — confirmed by v1.1 alone but
   reinforced here by the new seedB measurements showing the same
   spreads.
3. **The benchmark is capability-bound at this model size.** 64
   problems (39%) are solved by *no* condition we tested. The oracle
   union of all five conditions in v1.1 was 100/164 (61%). Adding the
   two new seedB conditions raises the all-conditions oracle union to
   about 105/164 (~64%). The remaining 60 problems appear to be
   genuinely beyond Qwen3.5-0.8B's reach at this configuration —
   throwing more samples or more priming will not recover them.

The optional `ttt_lowT_run2` / `ttt_highT_run2` conditions were not
run, because the v1.2 design doc gated them on Q1 returning "priming
is real." Q1 returned "priming is noise," so TTT's own internal noise
floor is moot — there is no priming signal to compare it against.

## 11. What v1.2 does *not* close

These are open questions that v1.2 explicitly did not address; future
work would have to:

- **Different model sizes.** All of this was Qwen3.5-0.8B. TTT could
  conceivably matter more for a different capacity class. The current
  v2 candidate of "just sample more at T=0.8" should be re-validated
  if anyone scales up.
- **Different benchmarks.** HumanEval is a small Python-coding set
  with very strict test-case validation. TTT might bite differently on
  tasks with looser correctness criteria or longer-horizon reasoning.
- **Different LoRA scales.** `r=4` on `q_proj` + `v_proj` is a small
  adapter. A much larger adapter or more update steps might extract
  signal that this configuration could not. v1.2 closes the
  *as-configured* TTT-search question; it does not close the entire
  TTT idea space.
- **Better selection rules.** All Q2 numbers are oracle unions (best
  candidate per problem by verifier). Real-world ensembling needs a
  selection rule without an oracle. The oracle figures are upper
  bounds; the practical numbers would be lower.

## 13. How to reproduce v1.2 from scratch

Assuming v1.1's `results/v11_full/detailed.json` already exists:

```bash
# 1. Generate the two seedB conditions (~2 hours)
python run_experiment.py \
    --num-problems 164 \
    --conditions base_temp_lowT_seedB,base_temp_highT_seedB \
    --output-dir results/v12_full \
    --skip-analysis

# 2. Run the v1.2 analyzer over v1.1 + v1.2 results (~1 second)
python analyze_v12.py
```

To override paths:

```bash
python analyze_v12.py \
    --v11 path/to/v1.1/detailed.json \
    --v12 path/to/v1.2/detailed.json \
    --out path/to/analysis_v12.txt
```

If you want to also measure TTT's internal noise floor (the optional
re-runs mentioned in the design doc §3), you would add new condition
configs to `CONDITIONS` in `run_experiment.py` that pass a different
master `torch.manual_seed` offset to `ttt_search.generate`. v1.2 did
not implement those — Q1's "priming is noise" verdict means the
internal-TTT noise floor is no longer relevant for deciding what to do
next.

## 14. Why this README exists

v1.2 is small in code and large in consequence. Two new condition
registrations, one new analyzer file, ~60 lines of substantive logic
— and the entire v1.1 verdict gets overturned. The headline "H1 + H2
HOLD" became "Whole direction dead. Just sample more." once the right
control was measured.

That swing makes one thing important to preserve next to the code:
**the threshold numbers** (`Q1_REAL_GAP=8`, `Q1_NOISE_BAND=5`,
`Q2_MIXED_HELPS=4`, `Q2_TIE_BAND=2`). They were chosen in the design
doc without reference to the data. If someone later argues the verdict,
the argument should be about those constants, not about the analyzer or
the run. They live in `analyze_v12.py` and are quoted verbatim in §5
of this document.

The other thing worth preserving is what the data shows beyond the
verdict line: **pure_highT_n8 at 54.9% is the strongest single n=8
configuration measured**, which is the practical recommendation for
anyone wanting more solved problems out of this same model on this
same benchmark. Sampling more at T=0.8 is the actionable conclusion
v1.2 supports; everything else has been ruled out.
